# ADR-005: Why Repository Pattern

## Context
The Nova Sphere codebase has suffered from tight coupling between the UI (Next.js components), the API routing logic, and the data access logic (direct Prisma calls or in-memory Maps). This makes unit testing impossible without an active database and heavily complicates migrations.

## Decision
We will strictly enforce the **Repository Pattern** and a layered architecture across the entire application:
`UI -> API Route / Server Action -> Application Service -> Repository Interface -> Prisma Implementation -> Database`

## Rationale
- **Separation of Concerns**: UI components handle display. Services handle business rules. Repositories handle data persistence.
- **Testability**: Repositories can be cleanly mocked in unit tests, allowing us to test complex business logic without a database.
- **Migration Safety**: We can seamlessly transition from the current in-memory Maps to Prisma by simply swapping the Repository implementation (`InMemoryProductRepository` -> `PrismaProductRepository`) without touching a single line of UI or Service code.
- **Portability**: If we ever migrate away from Prisma, we only rewrite the Repository layer.

## Consequences
- Increased boilerplate and structural complexity.
- Requires strict code-review governance to ensure developers do not bypass the Repository and call Prisma directly from UI components or API routes.
