# ADR-005: Event Sourcing & Transactional Outbox for Distributed Consistency

## Context
As Nova Sphere introduces complex distributed workflows (e.g., placing an order must update inventory, trigger an email, alert fraud detection, and update AI vectors), relying on a single massive SQL transaction across multiple domains leads to unacceptable locking, timeouts, and fragile monolithic logic.

## Decision
We decided to adopt the **Transactional Outbox Pattern** combined with **Event Sourcing** for cross-domain orchestration.
1. **OutboxEvent Table**: When a domain emits an event (e.g., `OrderPlaced`), it is written to the `OutboxEvent` table in the *same transaction* that writes the primary data (the `Order`).
2. **Background Processors**: A separate worker process polls the `OutboxEvent` table and publishes the events to downstream consumers, marking them as `PROCESSED`.
3. **OrderTimeline**: The Commerce domain will specifically use an `OrderTimeline` table to track every discrete state transition an order goes through, acting as an immutable event source for order history.

## Rationale
- The Outbox pattern guarantees that we never suffer from "Dual Write" failures (e.g., writing the order to the database succeeds, but sending the Kafka/Redis event fails, leaving the system inconsistent).
- `OrderTimeline` provides immediate customer-facing transparency ("Who cancelled my order and why?") without complex JOINs across fragmented logging tables.
- This paves the way for a message broker (Kafka or RabbitMQ) in the future, as the outbox table acts as a perfect buffer.

## Consequences
- Requires background worker processes to be constantly running.
- We must implement idempotent event consumers, as the outbox may occasionally retry a message delivery in failure scenarios ("At Least Once" delivery semantics).
