# ADR-002: Why Redis

## Context
Nova Sphere features real-time components (e.g., Auctions, Shopping Cart synchronization), high-frequency read patterns (Product Catalog, Personalization), and short-lived ephemeral state (AI Gateway rate limits, Semantic caching). Relying solely on PostgreSQL for these operations introduces unacceptable latency and database load.

## Decision
We will adopt **Redis** (managed via Upstash or standard Redis instances) as the primary in-memory data grid, caching layer, and pub/sub broker.

## Rationale
- **High-Frequency Writes**: Shopping cart modifications and auction bidding streams can easily overwhelm relational databases. Redis handles this seamlessly before eventually persisting the finalized state to PostgreSQL.
- **Rate Limiting**: Essential for the AI Gateway to prevent budget exhaustion and abuse.
- **Semantic Caching**: AI Gateway responses can be cached via Redis, significantly lowering API costs for repeated prompt queries.
- **Pub/Sub**: Essential for WebSocket communication, distributing real-time auction state, and signaling updates across serverless Edge nodes.

## Consequences
- Introduces eventual consistency for cached elements.
- Requires careful cache invalidation strategies (e.g., Cache-Aside pattern).
- State stored in Redis must be considered volatile unless backed by persistent storage.
