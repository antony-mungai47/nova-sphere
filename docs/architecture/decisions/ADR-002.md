# ADR-002: Prisma Migration Staging Strategy

## Context
The V3 Data Model introduces over 40 new tables across Commerce, AI, and Operations domains. Applying all 40 tables in a single Prisma migration (`npx prisma migrate dev`) presents a massive risk to the production database (Neon). A single failure could leave the database in an inconsistent state, and rolling back a giant schema change is highly complex.

## Decision
We decided to split the monolithic V3 schema update into three sequential migration stages:
1. **Stage 1**: Core Commerce (Orders, Inventory, Warehouse, Finance, Tenants).
2. **Stage 2**: AI Platform (ShoppingMemory, Vectors, AIBudget, Prompts).
3. **Stage 3**: Operations & Governance (AuditLogs, Deployments, Incidents).

## Rationale
- **Isolation**: Each domain operates somewhat independently. If the AI migration fails, the Commerce engine is unaffected.
- **Reviewability**: Smaller migrations are much easier to review in PRs.
- **Rollback Safety**: We can roll back specific domains without nuking the entire architecture.
- **Performance**: Neon branching and shadow DB operations succeed more reliably when processing smaller schema diffs.

## Consequences
- We must manually construct and sequence the `schema.prisma` files for each migration run.
- Referential integrity between domains (e.g., an Incident referencing an Order) must rely on loose coupling (string IDs) rather than strict foreign keys until both stages are deployed, or we must ensure Stage 3 strictly depends on Stage 1.
