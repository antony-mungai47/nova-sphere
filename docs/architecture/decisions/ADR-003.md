# ADR-003: Strict Append-Only Audit Trails & Soft Deletes

## Context
In a multi-vendor marketplace handling compliance, financial transactions, and ML scoring, mutating records destructively causes irrecoverable loss of historical context and violates basic financial governance.

## Decision
We mandate two strict rules for state mutation across the entire platform:
1. **Audit Logs are Append-Only**: The `AuditLog` table will contain a `createdAt` timestamp but no `updatedAt` field. Existing logs can never be mutated or deleted.
2. **Soft Deletes by Default**: Destructive operations (`DELETE FROM ...`) are strictly prohibited for core business entities (Orders, Inventory, Users, Products). Instead, we use `status` (e.g., "CANCELLED", "ARCHIVED") or `isActive = false`.

## Rationale
- Soft deletes allow for "undelete" operations, minimizing the blast radius of administrative errors.
- Append-only event tracking allows us to historically replay state or audit malicious actor behavior.
- ML models (like our Fraud engine) rely on accurate historical states. Deleting a fraudulent user deletes the training data required to catch the next one.

## Consequences
- Queries must explicitly filter by `isActive = true` or `status != 'DELETED'`, which adds slightly more boilerplate to repository queries.
- Storage costs will grow continuously, necessitating a long-term data archival strategy (e.g., moving cold data to S3 after 7 years) in Phase 15.
